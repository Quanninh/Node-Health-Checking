\begin{abstract}
Distributed systems depend on multiple interconnected nodes to communicate and cooperate reliably.
However, node failures, unstable connections, and disconnections can affect system availability if they are not detected and handled properly.
Many node health checking systems use a centralized approach because it is easier to implement, but this design creates a single point of failure when the central entity becomes unavailable.
This report presents a decentralized node health checking system that detects and repairs node and connection problems without relying on a central controller.

The proposed system uses a decentralized structure in which nodes monitor one another using phi-accrual failure detection.
Each node calculates a phi value and compares it with a predefined threshold to determine whether another node or connection is suspicious or failed.
When a problem is detected, the system applies repair and rewiring algorithms to restore the network topology.
Each node attempts to maintain a fixed number of neighbors, denoted as \textit{k}, and new nodes can join the network through a joining algorithm.

The system was implemented in Java using Spring Boot.
PostgreSQL and a dashboard were used only for visualization and observation, while the health checking mechanism can operate independently through terminal output.
Testing was performed with up to eight nodes, covering joining, termination, disconnection, and unstable connection scenarios.
Overall, the system demonstrates that decentralized node health checking can reduce dependency on a central entity and improve fault tolerance.

\end{abstract}

\section{Introduction}
\label{sec:introduction}
Distributed systems consist of multiple nodes connecting to each other and working with each other to perform tasks and provide reliable services.
In such systems, the readiness of a node is very important because individual nodes may become disconnected, unresponsive, or fail completely.
If these failures are not detected and handled quickly, they may affect communication between nodes and reduce the reliability of the system.

Node health checking is the process of monitoring if nodes are still active and responsive in the network.
In this project, health checking focuses on detecting the up or down status of each node through communication between connected nodes.
Based on the detected condition of the network, the system can identify healthy nodes, suspicious nodes, and failed nodes.
When a node or connection problem occurs, the system applies algorithms to remove failed nodes or unstable connections and reorganize the remaining connections to maintain progess of the network.

\subsection{Background}
\label{subsec:background}

A node health checking system is used to observe the status of nodes in a distributed system.
A node may be considered healthy when it can respond to health check messages within an acceptable time, and unhealthy when it stops responding, becomes unreachable, or behaves abnormally due to network or system issues.
The common centralized approach uses a dedicated monitoring component to check the status of all nodes.
In this model, nodes usually report their status to a central service, or the central service periodically sends health check requests to each node.
The centralized model has several advantages.
It is easier to design, easier to visualize, and simpler to maintain because all health information is collected in one place.
However, this also means that the reliability of the whole monitoring mechanism depends heavily on the central component.
If the central monitor fails, becomes disconnected, or is overloaded, the system may no longer be able to detect failures correctly.
In contrast, a decentralized node health checking system distributes the monitoring responsibility among the nodes themselves.
Each node maintains relationships with a limited number of neighboring nodes and performs health checks on those neighbors.
This creates a distributed monitoring structure where no single node is responsible for the entire system.
The relationship between nodes becomes an important part of the design because it determines how monitoring responsibility is shared, how failures are detected, and how the system can recover when connections are lost.

\subsection{Motivation}
\label{subsec:motivation}

The main motivation for this project is the single point of failure problem found in traditional centralized node health checking systems.
While centralized monitoring is practical for small or simple systems, it becomes less reliable when the central component is unavailable.
A failure in the monitoring service can prevent the system from detecting node failures, even though the actual distributed nodes may still be capable of communicating with one another.
A decentralized design reduces this dependency by allowing nodes to monitor each other directly.
This improves fault tolerance because the failure of one monitoring node does not stop the entire health checking process.
It also improves availability because the system can continue operating as long as enough node relationships remain active.
By distributing the health checking responsibility, the system avoids relying on a single central authority and becomes more resilient to partial failures.
This project is therefore motivated by the need for a node health checking mechanism that can continue functioning even when some components fail.
The design focuses on maintaining node relationships, detecting unhealthy nodes, and supporting recovery without depending entirely on a centralized monitor.

\subsection{Reasearch Gap}
\label{subsec:research_gap}

Existing decentralized monitoring approaches effectively detect node failures through heartbeat- or gossip-based mechanisms. 
However, many solutions focus primarily on failure detection and information dissemination, while topology maintenance and autonomous recovery are addressed as separate concerns. 
For example, Das et al. \cite{dasSWIMScalableWeaklyconsistent2002} designed SWIM, a scalable membership and failure detection protocol that efficiently disseminates failure information in distributed systems. 
Nevertheless, the protocol does not provide an integrated mechanism for autonomous topology repair following node failures. 
Consequently, limited attention has been given to unified frameworks that combine dynamic membership management, failure detection, gossip dissemination, and self-healing topology repair within a single decentralized architecture.

\subsection{Contributions}
\label{subsec:contribution}

This project presents a decentralized node health-checking system designed to reduce the single point of failure problem found in traditional centralized monitoring approaches.
Instead of depending on one central monitor to observe the entire system, the proposed design distributes the health-checking responsibility among the nodes themselves.
Each node monitors a limited number of neighboring nodes and participates in maintaining the overall health-checking structure.

The main contributions of this project are as follows:
\begin{itemize}
\item \textbf{Decentralized node health-checking design:}

The project proposes a neighbor-based monitoring structure where each node is responsible for checking the health status of a small set of other nodes.
This avoids relying on a single centralized monitor for failure detection.

\item \textbf{Node relationship and membership management:}

The system maintains relationships between nodes so that each node can monitor and be monitored by others.
When a node joins, fails, or becomes unreachable, the system attempts to update the node relationships to preserve the health-checking structure.

\item \textbf{Failure detection mechanism:}

The system implements a health-checking mechanism that allows nodes to detect unreachable or failed neighbors based on their responses.
This supports the identification of crash failures, temporary disconnections, delayed responses, and recovery situations.

\item \textbf{Repair and rewiring support:}

When a monitored node becomes unavailable, the system attempts to repair the affected node relationships by replacing failed or unreachable neighbors.
This helps the system continue operating even when individual nodes fail.

\item \textbf{Optional visualization support:}

The project includes a server application and dashboard for observing node status and failure events.
However, this component is used only for visualization and does not act as the central authority for the health-checking logic.
Therefore, the decentralized health-checking process can continue even without the dashboard or database.

\item \textbf{Experimental evaluation under failure scenarios:}

The system is evaluated using several scenarios, including normal operation, node joining, node crash, temporary network loss, unstable connection, node recovery, and dashboard or database unavailability.
These experiments demonstrate whether the system can detect failures and maintain node relationships without depending on a single centralized monitor.

\end{itemize}

Overall, the contribution of this project is a working prototype of a decentralized node health-checking system that improves fault tolerance and availability by distributing monitoring responsibility across multiple nodes. The system demonstrates how node relationships, failure detection, and recovery mechanisms can be combined to reduce dependency on a central monitoring component.


\section{Problem Statement}
\label{sec:problem_statement}
Many health checking systems rely on a centralized structure \cite{1Server}\cite{ApacheHadoop335}\cite{MonitorNodeHealth}, where a single central entity collects information and makes decisions for the entire network.
Although this approach is simple to manage, it may introduce a bottleneck and a single point of failure.
To address this limitation, this project adopts a decentralized approach.
Instead of depending entirely on one central controller, nodes participate in monitoring one another and responding to failures.
A dashboard may be used as a visual aid to observe the system state, but the health checking and recovery logic remain decentralized.

\subsection{Failure detection}
Failure detection is one of the fundamental problem for fault tolerance in distributed system because node may crash in any time. 
Traditional binary failure detectors classify a node only as "alive" or "unreachable" based on whether it responds within a timeout.

Although this model is simple, it is not flexible enough for dynamic distributed environments as this can cause false positives when a 
healthy node responds late due to network congestion, packet loss, high CPU usage, or temporary connection problems \cite{chandraUnreliableFailureDetectors1996}.

\subsubsection{\texorpdfstring{$\phi$}{phi} Accrual Calculation}

Hayashibara et al.~\cite{hayashibaraAccrualFailureDetector2004} proposed the \texorpdfstring{$\phi$}{phi} accrual failure detector to improve the flexibility of failure detection.
Instead of returning a boolean result, such as ``alive'' or ``unreachable'', the detector outputs a continuous suspicion level.
This allows the system or developer to interpret the monitoring information using different thresholds depending on the required trade-off between fast detection and false-positive reduction. 
A lower threshold can detect failures more quickly, while a higher threshold can reduce incorrect failure decisions. 
Experimental results from the original work show that the $\phi$ accrual failure detector performs comparably to other adaptive failure detection mechanisms, while providing greater flexibility for applications \cite{hayashibaraAccrualFailureDetector2004}.

\subsubsection{SWIM protocol}

Traditional heartbeating protocols may also suffer from scalability problems and false-positive failure detection. 
If every node monitors every other node, the number of heartbeat messages increases rapidly as the system grows.
In addition, if a node misses a heartbeat only because of temporary network problems, it may incorrectly mark another node as failed. 
Das et al.~\cite{dasSWIMScalableWeaklyconsistent2002} proposed SWIM to address these issues by separating failure detection from membership dissemination and using efficient peer-to-peer probing. 
In each protocol period, a node selects one target node from its membership and sends a direct ping. 
If the target replies with an acknowledgement, it is considered reachable. 
If no acknowledgement is received within the timeout, the probing node does not immediately declare the target as failed. 
Instead, it asks several helper nodes to perform indirect probes to the same target. 
If any helper node receives an acknowledgement from the target, it forwards this information back to the original probing node, showing that the target is still alive.

This indirect checking mechanism reduces false-positive failure detection because the decision is not based on only one communication path. 
A node is considered failed only when both the direct probe and the indirect probes fail. 
SWIM also disseminates membership changes using an infection-style, or gossip-style, approach, allowing updates about joins, leaves, suspicions, and failures to spread through the system with low per-node message overhead.

\subsection{System Design}

System architecture is another important design consideration. 
A common approach for coordinating distributed applications is to introduce a dedicated coordination service, such as ZooKeeper \cite{huntZooKeeperWaitfreeCoordination}. 
ZooKeeper is a replicated coordination service consisting of multiple servers, one of which acts as the leader. 
Clients may connect to any server in the ensemble, where read operations are served locally while write operations are forwarded to the leader, ordered, and replicated across the remaining servers.

This architecture provides strong consistency guarantees, high availability, and good scalability for read-dominated workloads. 
However, it also introduces a logically centralized coordination layer that must be maintained separately from the application nodes. 
This increases deployment and operational complexity and makes membership management dependent on a dedicated coordination ensemble.

For this reason, our system adopts a decentralized design instead. 
Rather than relying on an external coordination service, membership management, failure detection, and topology maintenance are distributed across the participating nodes themselves.

\subsection{Failure Detection: Hybrid SWIMPHI}
\label{subsec:hybrid_swimphi}
The proposed node health-checking system is influenced by both SWIM and the $\phi$ accrual failure detector. 
From SWIM, the system adopts the idea of decentralized peer-to-peer checking and helper-based indirect probing. 
This allows more than one node to participate in checking the status of a target node and reduces dependence on a single monitoring component. 
From the $\phi$ accrual detector, the system adopts the idea of using timing history and a continuous suspicion value instead of making an immediate binary decision after one missed response. 
In the proposed design, a target node is first checked through heartbeat or ping messages. 
If the target does not respond, helper nodes from the neighbor list can be used to verify whether the problem is a real node failure or only a local communication problem. 
After that, the $\phi$ value is used to evaluate the seriousness of the failure suspicion based on the response-time history.

By combining these two ideas, the proposed approach aims to improve both scalability and accuracy. 
SWIM contributes a scalable architecture for distributed membership checking, while the $\phi$ accrual detector contributes an adaptive decision mechanism for handling delay and message loss. 
Therefore, the system can avoid immediately deleting a node after a single missed heartbeat and can instead move through health states such as healthy, suspected, and failed based on both cooperative probing and suspicion-level evaluation.

As illustrated in Fig.~\ref{fig:phi_pipeline}, the algorithm combines indirect heartbeat verification with the $\phi$ accrual failure detector.
The indirect heartbeat mechanism reduces false-positive failure detection by allowing neighbor nodes to verify a suspicious node, while the $\phi$ accrual detector provides an adaptive suspicion value based on recent response-time behavior.

\subsubsection{Indirect Heartbeat Verification}
\label{subsubsec:indirect_heartbeat_verification}

Each node maintains a neighbor list containing the nodes with which it has a direct bidirectional connection.
When a node wants to check the availability of one of its neighbors, it sends a ping message and waits for an acknowledgement.
For example, if node $B$ wants to check node $A$, node $B$ sends a direct ping request to node $A$.
If node $A$ replies within the expected time range, node $B$ records the response time and classifies node $A$ as reachable.
This successful response also becomes part of the historical response-time data used later by the $\phi$ accrual detector.

However, if node $B$ does not receive an acknowledgement, it does not immediately mark node $A$ as failed.
Instead, the system uses indirect heartbeat verification. Specifically, node $B$ requests other nodes in its neighbor list to verify node $A$, such as nodes $C$ and $E$, to ping node $A$ on its behalf.
If any helper node receives an acknowledgement from node $A$, it notifies node $B$, and node $A$ is treated as alive.
In this case, the problem is considered a possible connection issue between nodes $A$ and $B$ rather than a complete node failure.

If neither the direct ping nor the indirect pings receive a response from node $A$, the system has stronger evidence that node $A$ may be unreachable. 
At this stage, the system does not rely only on the missing acknowledgements. Instead, it applies the $\phi$ accrual failure detector to calculate a suspicion value based on recent response-time history.

\subsubsection{\texorpdfstring{$\phi$}{phi} Accrual Calculation}
\label{subsubsec:phi_accrual_calculation}

The $\phi$ accrual detector avoids making a binary alive-or-dead decision based solely on a fixed timeout. Instead, it calculates a continuous suspicion value, $\phi$.

To make this decision, the detector relies on historical response-time data to determine whether the current waiting time is normal or abnormal. Each time a ping is sent and an acknowledgement is successfully received, the system records the response time. 
These recorded samples represent the recent communication behavior between nodes.

To keep the detector adaptive, response-time samples are stored in a sliding window. This window operates like a queue: when a new response-time sample is recorded, it is added to the window. If the window has already reached its maximum size, the oldest sample is removed. 
As a result, the window always contains the most recent response-time samples.

Let the response-time samples in the sliding window be:

\begin{equation}
X = \{x_1, x_2, x_3, \ldots, x_n\}
\end{equation}

The mean response time and standard deviation are calculated as:

\begin{equation}
\mu = \frac{1}{n}\sum_{i=1}^{n}x_i
\end{equation}

\begin{equation}
    \sigma = \sqrt{\frac{1}{n}\sum_{i=1}^{n}(x_i - \mu)^2}
\end{equation}

where $n$ is the number of samples, $x_i$ is a recorded response time, $\mu$ is the average response time, and $\sigma$ represents response-time variation.
These values are then used to estimate the response-time distribution for the $\phi$ calculation.

After the response-time distribution is estimated from the sliding window, the system calculates how unusual the current waiting time is.
Let $t$ be the current waiting time since the ping was sent.
The probability that a normal response time is greater than $t$ is defined as:

\begin{equation}
P(T > t) = 1 - F(t)
\end{equation}

where $F(t)$ is the cumulative distribution function of the response-time distribution.
The suspicion value is then calculated as:

\begin{equation}
\phi(t) = -\log_{10}(P(T > t)) = -\log_{10}(1 - F(t))
\end{equation}

A higher $\phi$ value means that the current waiting time is more unusual compared with recent response-time history, so the target node is more likely to be unreachable.

\subsubsection{Node State Classification}

After the $\phi$ value is calculated, the system compares it with predefined thresholds. 
These thresholds are configurable by the developer depending on how sensitive the failure detection process should be. 
A lower threshold can detect unreachable nodes faster, but may increase false positives. 
A higher threshold can reduce false positives, but may delay failure classification.

FINDING NEMO, short for \textit{Finding Neighbors and Evicted Members of their Own}, is the neighbor discovery and network joining algorithm proposed in this work. 
The name reflects the fact that a joining node must identify both prospective neighbors and any existing members that are evicted by those neighbors during the joining process, as discussed later in this section.

\begin{enumerate}

\item \textbf{Neighbor List}: A local data structure maintained by each node that stores information about its neighbors, such as their network addresses. The neighbor list has a fixed capacity of $k$, where $k$ is an even number. To maintain network connectivity and resilience, each node attempts to keep its neighbor list as full as possible whenever sufficient peers are available.

\subsection{Handling False Positives}
\label{subsec:false_positives}
Although the proposed approach significantly reduces false-positive detections, rare cases may still occur under severe network congestion or temporary network partitions, where the health-checking process is disrupted. 
In such situations, a healthy node may be incorrectly classified as \textbf{UNREACHABLE}.

To maintain consistency of the network topology, nodes classified as \textbf{UNREACHABLE} are removed from the active membership view and treated as departed members. 
Once connectivity is restored, the affected node is required to rejoin the network through the standard membership protocol, thereby rebuilding its neighbor relationships and restoring its participation in the system.

In this prototype, automatic process recovery is considered a deployment-level concern rather than a responsibility of the distributed protocol itself. 
The protocol only determines when a node should be considered unreachable and subsequently rejoin the network. In a production environment, this mechanism can be complemented by external supervisors such as systemd, Docker restart policies, Kubernetes controllers, or watchdog services to automatically restart failed node processes.
