\section{Introduction}
\label{sec:introduction}

Distributed systems consist of multiple nodes connecting to eachother and working with eachother to perform tasks and provide reliable services.
In such systems, the readiness of a node is very important because individual nodes may become disconnected, unresponsive, or fail completely.
If these failures are not detected and handled quickly, they may affect communication between nodes and reduce the reliability of the system.
\newline

Node health checking is the process of monitoring if nodes are still active and responsive in the network.
In this project, health checking focuses on detecting the up or down status of each node through communication between connected nodes.
Based on the detected condition of the network, the system can identify healthy nodes, suspicious nodes, and failed nodes.
When a node or connection problem occurs, the system applies algorithms to remove failed nodes or unstable connections and reorganize the remaining connections to maintain progess of the network.
\newline

Many health checking systems rely on a centralized structure, where a single central entity collects information and makes decisions for the entire network.
Although this approach is simple to manage, it may introduce a bottleneck and a single point of failure.
To address this limitation, this project adopts a decentralized approach.
Instead of depending entirely on one central controller, nodes participate in monitoring one another and responding to failures.
A dashboard may be used as a visual aid to observe the system state, but the health checking and recovery logic remain decentralized.

\subsection{Failure Models}
\label{subsec:failure_models}
Explicitly define the failure models considered in this report:
\begin{itemize}
    \item \textbf{Crash failure:} a node stops executing and no longer sends responses.
    \item \textbf{Network partition:} a node is alive but temporarily unreachable from one or more monitors.
    \item \textbf{Message loss:} heartbeat or health-check messages are dropped by the network.
    \item \textbf{Delay failure:} a node responds too slowly because of load, garbage collection, or network congestion.
    \item \textbf{Recovery:} a failed or unreachable node becomes healthy again and must rejoin the active membership view.
\end{itemize}

\subsection{Specific Algorithm}
\label{subsec:algorithm}
Describe the concrete health-checking algorithm used by the system. Include the heartbeat interval, timeout threshold, retry count, state transition rules, and membership update policy.

\begin{enumerate}
    \item Each node or monitor periodically sends a health-check request.
    \item A successful response marks the target node as healthy and refreshes its last-seen timestamp.
    \item A missed response increases the target node's suspicion counter.
    \item A node is marked suspected only after the configured retry threshold is reached.
    \item A suspected node is marked failed if it remains unreachable after the failure timeout.
    \item A failed node is restored to healthy after it responds successfully for a configured number of checks.
\end{enumerate}
